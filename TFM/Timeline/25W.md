## Padded points
entran a la loss si y solo si son marcados como unmatched o matchable. But what is point of matching keypoints of 0 descriptors???

Also there is kind of a bug. In the cache loader they are loading with random instead of zeros.

  For your two matchers:

  1. depth_matcher (dense depth)

  - Padded keypoints can still become -1 negatives if they
    get valid sampled depth and satisfy negative condition.
  - So they can contribute to negative loss.
  - See gluefactory/geometry/gt_generation.py:265,
    gluefactory/geometry/gt_generation.py:292, gluefactory/
    geometry/gt_generation.py:302.

  2. sparse_depth_matcher (your sparse map path)

  - You initialize as ignore and only promote with valid
    sparse masks/ids.
  - With Endomapper padding (valid_depth_mask=False,
    valid_3D_mask=False, point3D_ids=-1), padded points
    should stay ignored (-2) and not enter loss.
  - See gluefactory/geometry/gt_generation.py:85,
    gluefactory/geometry/gt_generation.py:100, gluefactory/
    geometry/gt_generation.py:105, gluefactory/geometry/
    gt_generation.py:151.
  - And Endomapper padding at gluefactory/datasets/
    endomapper.py:480, gluefactory/datasets/
    endomapper.py:484, gluefactory/datasets/
    endomapper.py:487.

  About cache_loader random descriptor padding vs SIFT zero
  padding:

  - Yes, there is inconsistency:
      - cache loader pads descriptors with random:
        gluefactory/models/cache_loader.py:26
      - sift pads descriptors with zeros: gluefactory/
        models/extractors/sift.py:303
  - They are not equivalent in practice, even if many padded
    points are ignored in GT:
      - Padded descriptors still affect matcher softmax
        normalization and attention context, so they can
        influence real matches indirectly.
      - This can change training dynamics and inference
        behavior.

  If you want consistent behavior, set cache descriptor
  padding to zeros too (or add an explicit valid-keypoint
  mask and use it in matcher/loss).
