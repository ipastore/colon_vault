```bash
def scaled_dot_product_attention(query, key, value, attn_mask=None, dropout_p=0.0, is_causal=False, scale=None) -> torch.Tensor:
    # Efficient implementation equivalent to the following:
    L, S = query.size(-2), key.size(-2)
    scale_factor = 1 / math.sqrt(query.size(-1)) if scale is None else scale
    attn_bias = torch.zeros(L, S, dtype=query.dtype)
    if is_causal:
        assert attn_mask is None
        temp_mask = torch.ones(L, S, dtype=torch.bool).tril(diagonal=0)
        attn_bias.masked_fill_(temp_mask.logical_not(), float("-inf"))
        attn_bias.to(query.dtype)

    if attn_mask is not None:
        if attn_mask.dtype == torch.bool:
            attn_mask.masked_fill_(attn_mask.logical_not(), float("-inf"))
        else:
            attn_bias += attn_mask
    attn_weight = query @ key.transpose(-2, -1) * scale_factor
    attn_weight += attn_bias
    attn_weight = torch.softmax(attn_weight, dim=-1)
```

```bash
# Efficient implementation equivalent to the following:
def scaled_dot_product_attention(query, key, value, attn_mask=None, dropout_p=0.0,
        is_causal=False, scale=None, enable_gqa=False) -> torch.Tensor:
    L, S = query.size(-2), key.size(-2)
    scale_factor = 1 / math.sqrt(query.size(-1)) if scale is None else scale
    attn_bias = torch.zeros(L, S, dtype=query.dtype, device=query.device)
    if is_causal:
        assert attn_mask is None
        temp_mask = torch.ones(L, S, dtype=torch.bool).tril(diagonal=0)
        attn_bias.masked_fill_(temp_mask.logical_not(), float("-inf"))

    if attn_mask is not None:
        if attn_mask.dtype == torch.bool:
            attn_bias.masked_fill_(attn_mask.logical_not(), float("-inf"))
        else:
            attn_bias = attn_mask + attn_bias

    if enable_gqa:
        key = key.repeat_interleave(query.size(-3)//key.size(-3), -3)
        value = value.repeat_interleave(query.size(-3)//value.size(-3), -3)

    attn_weight = query @ key.transpose(-2, -1) * scale_factor
    attn_weight += attn_bias
    attn_weight = torch.softmax(attn_weight, dim=-1)
    attn_weight = torch.dropout(attn_weight, dropout_p, train=True)
    return attn_weight @ value
```

### FP16 vs BF16

SDPA means scaled dot-product attention. In PyTorch, F.scaled_dot_product_attention(...) is the attention primitive that can dispatch to FlashAttention, memory-efficient attention, or a math fallback.

  Mathematically, for queries Q, keys K, and values V:

$$
  A = \text{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)
$$

$$
  
  \text{output} = AV
  
$$

  where:

  - $d$ is the key/query feature dimension
  - $QK^T$ gives pairwise similarity scores
  - dividing by $sqrt(d)$ keeps scores from growing too large
  - softmax turns scores into attention weights

  If you never use a mask, then your call is just:

  $\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V$

  Why precision matters:

  - attention is numerically sensitive because it uses matrix multiplies and then softmax
  - softmax amplifies differences in logits
  - if logits get too large or too tiny, low-precision formats can lose information

  **fp16 vs bf16**

  Both use 16 bits total, but they distribute them differently.

  **fp16:**
  - 1 sign bit
  - 5 exponent bits
  - 10 mantissa bits

  **bf16:**
  - 1 sign bit
  - 8 exponent bits
  - 7 mantissa bits

  Interpretation:
  - exponent controls dynamic range: how large or small numbers can be
  - mantissa controls precision: how finely numbers are represented

  So:
  - fp16 has better precision near a given scale
  - bf16 has much larger dynamic range

[[detailed_explanation_fp]]


  That is why bf16 is often more stable in deep learning: it can represent much larger and much smaller values before overflow/underflow.

  Why this matters in attention

  The risky part is the logits:
  
$$
  L = \frac{QK^\top}{\sqrt{d}}
$$


  If entries of Q and K are moderately large, then dot products can become large too. In fp16, large intermediate values are easier to overflow. In bf16, the larger exponent range makes that much less
  likely.

  Then softmax(L) computes:
$$
  \text{softmax}(L_i) = \frac{e^{L_i}}{\sum_j e^{L_j}}
$$
  

  This is numerically dangerous when some logits are much bigger than others:
  - very large positive logits can make exponentials blow up
  - very negative logits can collapse to zero
  - small differences between close logits may be lost due to rounding

  In practice, implementations stabilize softmax by subtracting the maximum logit first:

$$
  \text{softmax}(L_i) = \frac{e^{L_i - \max(L)}}{\sum_j e^{L_j - \max(L)}}
$$


  but low precision still affects:
  - the dot products
  - the subtraction
  - the exponentials
  - the final weighted sum AV

  Stability tradeoff

  fp16:

  - more mantissa bits, so slightly finer local precision
  - much smaller range, so more overflow/underflow risk
  - can be fast, but less robust

  bf16:

  - fewer mantissa bits, so rougher local precision
  - much larger range, so much safer for large activations
  - usually preferred for training stability on supported GPUs

  In modern training, range problems are usually more damaging than the small precision advantage of fp16. That is why bf16 is often the better default.

  For your case

  If:

  - you never use a mask
  - you are on CUDA
  - your GPU supports bf16 well

  then the best default is usually:

  - keep tensors in bf16
  - call scaled_dot_product_attention directly
  - do not force .half()

  Because .half() converts bf16 -> fp16, which usually reduces stability and may not improve speed enough to justify it.

  So this:

  args = [x.half().contiguous() for x in [q, k, v]]

  is probably worse for you than:

  args = [x.contiguous() for x in [q, k, v
